# 自动行驶小车（H题）—— 路线分解 & 代码适配

## 一句话结论

**四个任务用同一份底盘代码，只换路段序列**。就像你已有 `GrayRouteSegment_t` 枚举拼不同任务一样——巡弧线、直线冲、过点检测是共享的底层能力。

---

## 场地布局推断

根据题目描述（场地 ≥220×120cm，两个半圆弧 r=40cm，ABCD 四顶点）：

```
    A ════════════════ B      ← 半圆弧（半径 40cm，向上凸）
    │                   │
    │    220 × 120      │     ← 弧顶点距直边有多远由 pic1 定
    │                   │
    D ════════════════ C      ← 半圆弧（半径 40cm，向下凸）
```

A、B、C、D 是弧线顶点（弧与直边的切点），也是题目要求的过点检测位置。具体弧线的凸向和直边长度见 pic1 场地示意图。

---

## 路段类型

不管场地具体几何，能走的只有两种动作：

| 路段类型 | 传感器特征 | 控制策略 |
|----------|-----------|----------|
| **直线** | 灰度偏差稳定在中心附近 | 保持直行，速度 300 mm/s |
| **弧线** | 灰度偏差持续偏一侧（外侧传感器看到黑线） | 纯追踪曲率控制，降速 185~240 mm/s |
| **过点** | 8 路全黑 ≥7 个（十字/大面积黑线） | 触发直冲 + 位置记录 |

注意：H 题的 ABCD 四个顶点不是十字路口——场地除弧线外**不得添加标记**。过点检测不能靠全黑触发，要靠**"弧线尽头丢线"或"弧线→直线的过渡"**来识别。这和你的断桥入口检测逻辑类似。

## 策略：只做 T4，向下兼容

**T4 = 按任务(3)的路径跑 4 圈**，是难度最高、分值最大的（30 + 30 = 60 分）。

T4 跑通之后：

| 任务 | 做法 | 分值 |
|------|------|------|
| (3) | 同一条 `g_route[]`，计数器改成 1 圈 | 30 |
| (2) | 换一套顺时针路段序列，跑 1 圈 | 20 |
| (1) | 只跑第一段（A→B 弧线），到了就停 | 20 |

**不需要为每个任务写一套代码**——路段序列是数据，任务模式只是换数组 + 换停止条件。

## 目标路径：任务 (3) 的路线

设场地布局为：
- 顶弧 A↔B（逆时针：A→B = 弧、B→A = 弧反向）
- 右直 B↔C
- 底弧 C↔D
- 左直 D↔A

### 任务 (1)：A→B，≤15s

```
A ──[顶弧]──→ B（停）
```

最简单——沿一条弧线从 A 走到 B。但 A 和 B 都是弧端点，如何知道"到 B 了"？

**方案**：里程计。A→B 弧线长度 = π × 40cm ≈ 126cm。从 A 出发后累计编码器里程 ≥120cm 时减速，看到灰度恢复（弧线结束）→ 停车。

对应任务模式：`TASK_1_AB_STOP`（你已有的模式 0）

### 任务 (2)：A→B→C→D→A，≤30s

```
A ──[顶弧]──→ B ──[右直]──→ C ──[底弧]──→ D ──[左直]──→ A（停）
```

顺时针一圈。路段序列：

| 段 | 起点 | 终点 | 动作 |
|----|------|------|------|
| 1 | A | B | 巡弧线（左转弧） |
| 2 | B | C | 直线直行 |
| 3 | C | D | 巡弧线（右转弧） |
| 4 | D | A | 直线直行 |

对应你已有的 `TASK_2_CW_1LAP`。

### 任务 (3)：A→C→B→D→A，≤40s

```
A ──[左直]──→ D ──[底弧]──→ C ──[右直]──→ B ──[顶弧]──→ A（停）
```

看起来是"先往下走"，和任务 (2) 方向相反但也不完全一样。路段序列：

| 段 | 起点 | 终点 | 动作 |
|----|------|------|------|
| 1 | A | D | 直线直行 |
| 2 | D | C | 巡弧线 |
| 3 | C | B | 直线直行（反向） |
| 4 | B | A | 巡弧线 |

> 注：A→C 的路径到底是 A→D→C 还是 A→B→C，取决于 pic1 中弧线的位置。上面按"左直先下"假设，看 pic1 修正。

对应新模式：`TASK_3_CCW_1LAP`（逆时针一圈——你已有）。

### 任务 (4)：按 (3) 路径跑 4 圈

和任务 (3) 完全相同的路段序列，加一个 `lap_count` 计数器，跑到 4 圈停车。用时越少越好 → 每圈固定速度要快（300 mm/s 基础速度以上）。

对应你已有的 `TASK_4_CCW_4LAP`。

## 和你现有代码的对应关系

| 你的枚举 | H 题路段 | 说明 |
|----------|---------|------|
| `BRIDGE_TOP_LR` / `ARC_*` | A↔B（顶弧） | 弧线巡线 + 端点过点检测 |
| `ARC_*` | C↔D（底弧） | 同上，方向相反 |
| — | B↔C（右直） | 纯直线，灰度稳在中心 |
| — | D↔A（左直） | 同上 |

### 需要调整的地方

1. **过点检测逻辑**：现有代码依赖"8 路全黑 ≥7 个"触发十字/过点。H 题场地**不允许标记**——ABCD 四个顶点只能靠"弧线→直线"或"直线→弧线"的**过渡特征**来检测。需要新写一个 `Gray_DetectTransition()` 函数：连续 N 个周期灰度从"偏左"变成"居中"（或反之），判为弧线端点。

2. **里程计辅助**：弧线长度已知（π × 40cm ≈ 126cm），用编码器累计距离辅助判断"应该到弧线终点了"——即使灰度过渡没检测到，也能靠里程兜底。

3. **路段枚举扩展**：现有 `GrayRouteSegment_t` 的 8 个枚举值是为"有桥"赛道设计的（顶桥 LR、底桥 RL 等）。H 题没有断桥，弧线段就是单纯的弧，可以把枚举简化或映射到现有值。

4. **起终点定位**：ABCD 没有物理标记，靠**任务开始时的摆放位置**确定"哪个点是 A"。首次出发后，按路段序列 + 过点检测自动推进状态机。

### 不需要改动的地方

- 灰度加权质心 → 巡线偏差 → 曲率控制 ✅
- 增量式 PI 速度闭环 ✅
- 编码器里程计 ✅
- 任务状态机框架 ✅
- OLED / LED 状态指示 ✅

## 路段状态机伪代码

```c
void Gray_RouteTick(void)
{
    GraySegment_t seg = g_route[g_current_seg];

    switch (seg.type) {
    case SEG_ARC:
        // 巡弧线：灰度偏差算曲率
        Gray_LineFollow(seg.arc_speed);
        // 检测弧线→直线过渡
        if (Gray_DetectArcEnd()) {
            Gray_OnPointReached(seg.end_point);  // 声光提示
            g_current_seg++;                      // 推进到下一段
        }
        break;

    case SEG_STRAIGHT:
        // 直线直行
        Gray_StraightRun(seg.straight_speed);
        // 检测直线→弧线过渡（或里程计）
        if (Gray_DetectStraightEnd() || OdometerReached(seg.length_mm)) {
            Gray_OnPointReached(seg.end_point);
            g_current_seg++;
        }
        break;

    case SEG_STOP:
        Stop();
        TaskComplete();
        break;
    }
}
```

四个任务的唯一区别是 `g_route[]` 数组内容不同。任务切换时重新赋值路段序列即可。
