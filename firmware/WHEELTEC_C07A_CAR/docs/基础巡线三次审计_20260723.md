# 基础巡线三次审计：既有审计复核

日期：2026-07-23

## 当前控制链路复核

```text
5 ms TIMG0 中断
  -> 读取并清零左右编码器计数
  -> Run_Mode=1：依次选通 8 路灰度，计算黑线质心
  -> 纯追踪曲率换算为 Move_Z（负号）
  -> 差速逆解：左 = Vx - Vz*轮距/2，右 = Vx + Vz*轮距/2
  -> 左右增量 PI
  -> 两路 PWM 均取反后输出
```

灰度全白时，目标线速度和角速度都会清零；当前代码没有原地搜线和十字直冲。因此“丢线后不会固定单向旋转”的结论仍成立。

在 `45 mm/s`、最大横向偏差 `42.5 mm`、前视距离 `260 mm` 时，公式计算出的最大角速度约为 `0.095 rad/s`，低于 `0.40 rad/s` 限幅；现有角速度限幅在正常单线场景通常不会介入。也就是说，若仍发生猛烈转向，优先检查灰度线序/极性、电机或编码器方向，而非先怀疑该限幅。

## 必须保留为实车待证的事项

1. 文档声称“通道 0 在左、通道 7 在右”，但当前仓库未保存对应的原始实测记录。转向负号是否让车辆回到线中心，仍须通过逐路遮挡验证。
2. `2bfb268` 的左右 PWM 同时取反是一次实车接线修正；测速中左轮保持原始计数、右轮额外取反。闭环正负反馈只能用抬车试验确认，不能仅凭代码或旧工程推断。
3. 实际减速比必须在电机型号/实测脉冲数上确认。若硬件仍为 30:1，当前 `CPR=728` 会使速度估计偏高约 `7.1%`，PI 和显示都会随之失准。

## 新发现

### P1：编码器共享计数未声明为 `volatile`

`Hardware/encoder.c` 的 `Get_Encoder_countA/B` 在 GPIO 中断中更新、在 5 ms 定时器中读取并清零，但其定义和声明均为普通 `int`。这不保证优化编译下的跨中断可见性。应在后续代码修正时将声明和定义统一为 `volatile int`，并评估读取/清零是否需要临界区保护。

另外，同一端口两个相位若在一次 `GROUP1_IRQHandler()` 到来前都挂起，当前 `if ... else if ...` 只处理其中一个相位，却会清除两个挂起位；高转速下可能丢计数。基础低速验证影响较小，但不能把当前实现视为严格的 2 倍频计数器。

### P1：SysConfig 仍保留与灰度地址脚冲突的 CCD 配置

`empty.syscfg` 为 `ADC12_CCD` 建议分配 `PA27`，而生成的 GPIO 配置把同一引脚作为灰度 `GRAY_AD0` 输出。当前生成的 `ti_msp_dl_config.c` 未调用 `SYSCFG_DL_ADC12_CCD_init()`，所以基础灰度巡线运行时实际由 AD0 占用 PA27；但若重新生成 SysConfig，此冲突会造成配置不确定或重新暴露 CCD 配置问题。当前项目只做 8 路灰度时，应把 CCD 配置明确移出该 SysConfig 或迁移到不冲突的项目。

### P2：遥控 DMA 接收没有首次启动

`BT_DAMConfig()` 只在 DMA 完成或缓冲区重置时调用，启动流程没有首次调用；UART1 也只启用了 DMA 完成中断。故 `Run_Mode=0` 的蓝牙遥控路径不能假定可用。此项不影响 `Run_Mode=1` 的基础巡线，但与 `empty.c` 中的 `BTBufferHandler()` 意图不一致。

### P3：定时器中断判断应写成实际返回值比较

处理函数先读取 `DL_TimerA_getPendingInterrupt(TIMER_0_INST)`，随后使用恒为真的枚举常量 `if (DL_TIMER_IIDX_ZERO)`。外层非零判断使当前 ZERO 事件可以运行，因而不是本次方向问题的原因；但代码表达不完整，后续应保存返回值并显式比较 `DL_TIMER_IIDX_ZERO`。

## 复测顺序

1. 先停机逐路遮挡，记录每个物理传感器对应的 `Gray_Raw[0..7]`；确认黑线极性与左右顺序。
2. 抬车、以当前代码的中间线输入启动，确认两轮均为前进，且 OLED 上两轮实测速度均为正。
3. 分别在左、右侧制造黑线偏差，确认目标轮速差和实车转向均朝向黑线。
4. 方向正确后，从当前 `45 mm/s` 开始阶梯试跑；确认稳定后再提高基础速度与角速度上限。
5. 最后确认减速比；再处理 `volatile`、双相同时挂起和 SysConfig 冲突等工程可靠性问题。
