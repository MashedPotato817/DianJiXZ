## 1. 本次观察结论

检查 `pic/` 中 2026-07-23 的三张实拍帧及 `log.md` 后确认：当前画面叠加的圆、十字和四角落在窗框/窗格上，而不是可确认的靶纸。当前系统把背景中与 A4 靶纸长宽比相近的窗格持续判为 `detect`，因此“连续检测”不等于“正确识别靶纸”。

日志采样记录中有 `detect=572`、`hold=70`、`none=118`、`reject=28`。这些数字用于说明候选输出非常频繁，不能作为识别成功率。

## 2. 根因

当前 `cvlite_demo.py` 的入口是全画面 `cv_lite.grayscale_find_rectangles_with_corners()`；筛选条件只有候选矩形的长宽比和面积，并取最大面积候选。现有长宽比范围为 `0.52~0.92`，与靶纸及窗格都兼容；算法没有验证黑框、同心圆或红点等靶纸身份特征。

另外，当前仅采集灰度图，无法用红点做校验；`target_config.py` 也未被 `cvlite_demo.py` 导入，因此其中的红点阈值不参与当前流程。

时序逻辑亦需修正：连续确认阈值只影响缓存，不会阻止第一帧候选绘制和输出；跳变拒绝时只回退中心，不回退四角，导致 `reject` 日志中的中心与 corners 可能不属于同一候选。

## 3. 调研结论与方案决策

**明确结论：不要继续把“全画面矩形检测”作为主入口；应改为“黑框 ROI → ROI 内四边线 → 对角线交点 → 圆环/红点校验”的两阶段几何方案。**

该结论来自项目内已有调研：全画面 `find_rects()` 被判定会受局部边缘和背景干扰；“黑色连通域 ROI → ROI 内四边线 → 对角线交点”被列为推荐主方案。来源：选自 [靶心识别方案调研.md（第 14～18 行）](/C:/Users/zhntd/Desktop/DianJiXZ/firmware/K230_Target_Detection/research/靶心识别方案调研.md:14)。

推荐处理链路：

```text
RGB565 原始图像
  → 灰度黑色阈值 + find_blobs()，获得黑框候选 ROI
  → 仅在 ROI 内执行 find_lines()/find_line_segments()，恢复四条边
  → 相邻边求四角，两条对角线求交点，得到初始靶心
  → 依据四角做透视矫正
  → 在矫正图中校验同心圆；在中心小 ROI 检查红点
  → 连续确认后输出 valid、center_x、center_y
```

上述流程与现有靶心调研所列推荐实现路径一致。来源：选自 [靶心识别方案调研.md（第 20～27 行）](/C:/Users/zhntd/Desktop/DianJiXZ/firmware/K230_Target_Detection/research/靶心识别方案调研.md:20)。

### 为什么取对角线交点

靶纸是平面矩形，倾斜后投影为任意四边形。使用候选外接矩形中心会引入透视偏差；靶纸四角构成的两条对角线交点才对应其几何中心。若实物印刷圆心和纸张中心有固定偏移，应在正对相机时标定为常量补偿。来源：选自 [靶心识别方案调研.md（第 29～31 行）](/C:/Users/zhntd/Desktop/DianJiXZ/firmware/K230_Target_Detection/research/靶心识别方案调研.md:29)。

### cv_lite 的定位

保留 `cv_lite`，但只把它视作可选的性能加速工具，而不再单独负责靶纸身份判定。调研显示其底层是 Canny、轮廓拟合和矩形筛选，速度优势明显；但这类全画面矩形候选仍会遇到窗格等同形背景干扰。来源：选自 [2025-电赛E题-视觉方案调研.md（第 55～70 行）](/C:/Users/zhntd/Desktop/DianJiXZ/firmware/K230_Target_Detection/research/2025-电赛E题-视觉方案调研.md:55)。

## 4. 实施建议

### 阶段 A：先完成黑框 ROI

1. 保留 RGB565 原图，并从灰度图对暗色区域作阈值分割。
2. 使用 `find_blobs()` 找候选黑色连通域；按面积、长宽比、边界完整性筛选。
3. 候选 ROI 必须连续覆盖粗黑外框；窗格、桌面或背景黑物体不应直接成为有效靶。

本阶段验收标准来自现有调研：ROI 覆盖黑框；ROI 内能检测两近水平、两近竖直边；靶纸离开、黑框不完整或尺寸过小时输出 `valid=0`，不输出旧坐标。来源：选自 [靶心识别方案调研.md（第 32～34 行）](/C:/Users/zhntd/Desktop/DianJiXZ/firmware/K230_Target_Detection/research/靶心识别方案调研.md:32)。

### 阶段 B：四边几何与身份校验

1. 在 ROI 内检测边线，求相邻边交点得到四角，拒绝非凸、边长异常或方向不符合靶纸的候选。
2. 用两条对角线交点作为靶心初值。
3. 对四角做透视矫正，在统一坐标中检查黑色同心圆的数量、共心性和尺寸比例。
4. 在靶心附近的小 ROI 以 RGB 阈值检查红点；红点只用于确认或微调，不单独决定靶心。

全画面圆检测不作为主方案，因为靶纸倾斜时圆环会投影成椭圆；圆环应在已锁定靶框后作为二阶段校验。来源：选自 [靶心识别方案调研.md（第 14～17 行）](/C:/Users/zhntd/Desktop/DianJiXZ/firmware/K230_Target_Detection/research/靶心识别方案调研.md:14)。

### 阶段 C：时序与通信

1. 连续至少 3 帧同时通过“黑框 + 四边 + 圆环/红点校验”才发布 `valid=1`。
2. 跳变候选应整帧丢弃，恢复上一份完整有效状态或输出 `valid=0`；不得只回退中心而保留新 corners。
3. `hold` 只作为显示或极短暂平滑状态，不能作为控制系统长期使用的坐标来源。
4. UART 发送坐标时复用既有 `0xAA + x + y + area + XOR` 协议；无有效靶时令 `area=0`，使 MSPM0 明确区分“无目标”与“坐标 (0,0)”。来源：选自 [docs_learn/ques8.md（第 35～52 行）](/C:/Users/zhntd/Desktop/DianJiXZ/docs_learn/ques8.md:35)。

## 5. 采集与调参要求

- 相机避免直接面向窗户、灯管等强逆光背景；尽量锁定曝光和白平衡。
- 靶纸高度建议占有效画面约 1/4～1/2，黑框线宽至少保持 3～5 px。
- 建立包含靶纸远近、旋转、倾斜、强反光、空场、窗格、桌椅和局部遮挡的测试帧集。
- 每轮调参记录：黑色阈值、ROI 面积范围、边线阈值、圆环校验阈值、红点阈值、帧率和误检样例。

早期调研给出的二值化、形态学和面积阈值只能作为实机标定起点，不应直接视为通用固定值。来源：选自 [2025-电赛E题-视觉方案调研.md（第 30～53 行）](/C:/Users/zhntd/Desktop/DianJiXZ/firmware/K230_Target_Detection/research/2025-电赛E题-视觉方案调研.md:30)。

## 6. 验收目标

| 项目 | 目标 |
| --- | --- |
| 空场、窗格背景 | 不输出 `valid=1` |
| 靶纸正常入镜 | 连续有效率 ≥ 95% |
| 靶心稳定性 | 静止靶标抖动 ≤ 2 px |
| 靶纸离开/遮挡/黑框残缺 | 输出 `valid=0`，不沿用旧坐标作控制 |
| 性能 | 记录真实 FPS，并满足控制链路刷新需求 |

## 7. 是否训练模型

当前不训练 YOLO/KPU。靶纸的黑框、白底、圆环和红点具有固定几何结构，几何方法比检测框中心更适合输出高精度靶心。只有以下情况满足任一项时才进入模型方案：黑框经常遮挡；场地有大量相似黑色矩形且几何校验仍不能稳定排除；靶纸样式频繁变化。即使训练模型，模型也只用于给出 ROI，最终靶心仍由四角几何计算。来源：选自 [靶心识别方案调研.md（第 35～41 行）](/C:/Users/zhntd/Desktop/DianJiXZ/firmware/K230_Target_Detection/research/靶心识别方案调研.md:35)。

## 9. 显示与缩放优化方案（供 Claude Code 选择实施）

### 9.1 当前黑边的计算原因

当前链路不是“960P 检测后放大到 480P 显示”，而是：`1280×960`（日志中的当前相机/ISP 输出配置，4:3）→ `320×240` 灰度检测（横、纵均缩小 4 倍）→ `640×480` 显示画布。最后一步以 `(160, 120)` 坐标将 `320×240` 图像**原尺寸**显示；这不是缩放，而是居中贴图。因此四周保留黑边：左右各 `(640-320)/2=160 px`，上下各 `(480-240)/2=120 px`。来源：选自 [cvlite_demo.py（第 16～19、73～77、151～153 行）](/C:/Users/zhntd/Desktop/DianJiXZ/firmware/K230_Target_Detection/cvlite_demo.py:16)。

若用户所说的“黑框”是显示器物理边框，软件无法填充；若是上述视频外的软件黑边，以下方案均可消除或按比例缩小。

当前 `1280×960 → 320×240` 同为 4:3，不需要裁切且不发生非等比形变。代码未显式配置 `crop`；由于输入和输出宽高比相同，即使底层使用硬件缩放，也无需进行画面裁切。应避免未来在未明确裁切策略的情况下改成 `1920×1080 → 320×240`，因为二者分别是 16:9 与 4:3，可能引入裁切或几何变形。

### 9.2 方案 A：640×480 双通道硬件直显 + 320×240 检测（当前硬件首选）

适用条件：实际显示屏为 640×480，首要目标是保留摄像头完整 4:3 视场，同时保证检测性能。

```text
1280×960 相机/ISP 完整 4:3 视野
├─ ch0：640×480 YUV420 → Display.bind_layer(VIDEO1) → 全屏显示
└─ ch1：320×240 GRAYSCALE → cv_lite 靶纸检测 → OSD 叠加
```

两条通道均保留完整 4:3 视野，坐标映射严格为：

```text
x_display = 2 * x_detect
y_display = 2 * y_detect
r_display = 2 * r_detect
```

这不是把算法图像在 Python 中放大，而是让硬件分别提供显示和检测输出：检测仍为 `320×240=76800` 像素，算法负担不变；显示为 640×480 全屏，用户可更清楚地观察靶纸与叠加结果。`Display.bind_layer()` 可避免将 640×480 图像逐帧经 Python `resize` 后再 `show_image()`。

项目现有 `main.py` 已使用两个传感器通道和 `Display.bind_layer()`，可作为此方案的架构参考。来源：选自 [main.py（第 64～84 行）](/C:/Users/zhntd/Desktop/DianJiXZ/firmware/K230_Target_Detection/main.py:64)。

### 9.3 方案 B：仅放大当前检测画面（最小改动，不推荐作为最终版）

做法：保持 `320×240` 检测，额外将图像放大后显示。

| 显示目标 | 保比例计算 | 显示结果 |
| --- | --- | --- |
| 640×480（4:3） | `s=min(640/320,480/240)=2` | 正好得到 640×480，无黑边、无几何变形 |
| 1920×1080（16:9） | `s=min(1920/320,1080/240)=4.5` | 得到 1440×1080，左右各保留 240 px 黑边 |
| 1920×1080（裁剪铺满） | `s=max(1920/320,1080/240)=6` | 得到 1920×1440，裁掉上下各 180 px |

禁止采用 `sx=1920/320=6`、`sy=1080/240=4.5` 的非等比拉伸：它会将圆环拉成椭圆，误导对靶心与几何质量的人工判断。

优点：改动最少。缺点：每帧软件 `resize`/复制 1080P 图像会增加带宽和 CPU 开销，且并不能提高检测精度；只适合临时调试，不应作为最终视觉体验方案。

### 9.4 方案 C：4:3 保比例全高度显示（外接 HDMI 时保留完整视野）

做法：HDMI 输出设为 `1920×1080`，仍保留完整的 `320×240`（4:3）检测视野，以比例 `s=4.5` 显示为 `1440×1080`，视频水平居中。

```text
1920×1080 屏幕
┌────240────┬────────1440×1080 视频────────┬────240────┐
│  黑边     │       完整 4:3 相机视野        │   黑边    │
└───────────┴───────────────────────────────┴───────────┘
```

检测坐标映射为 `x_screen = 240 + 4.5*x_detect`、`y_screen = 4.5*y_detect`。该方案不丢失上下视野，且几何不失真；适合靶纸可能出现在画面上下边缘的调试场景。缺点是无法“铺满整个 16:9 屏幕”。

### 9.5 方案 D：16:9 全屏硬件预览 + 16:9 低分辨率检测（仅外接 HDMI 的全屏备选）

做法：使用两个传感器通道，而不是软件放大检测帧：

```text
传感器 1920×1080@60
├─ ch0：1920×1080 YUV420 → Display.bind_layer(VIDEO1) → HDMI 全屏视频
└─ ch1：320×180 GRAYSCALE / RGB565 → 靶纸检测 → OSD 叠加
```

`320×180` 与 `1920×1080` 同为 16:9，故坐标、长度和半径都按固定比例映射：

```text
scale = 1920 / 320 = 1080 / 180 = 6
x_screen = 6 * x_detect
y_screen = 6 * y_detect
r_screen = 6 * r_detect
```

优点：全屏、无黑边、无几何变形；视频由硬件通道直显，不需 Python 每帧缩放；检测像素数从当前 `320×240=76800` 降为 `320×180=57600`，理论上减少 25% 的逐像素检测负担。缺点：相对 4:3 视野会裁掉上、下区域；须确认靶纸不会长期出现在这些区域。

K230 官方示例采用同一传感器的全分辨率显示通道和低分辨率处理通道；显示通道通过绑定直接送至显示层。来源：选自 [CanMV 官方 Sensor 示例](https://www.kendryte.com/k230_canmv/en/main/example/media/sensor.html)。K230 的 LT9611 显示接口支持 `1920×1080@60`。来源：选自 [CanMV 官方 Display API](https://www.kendryte.com/k230_canmv/en/v1.3/api/mpp/K230_CanMV_Display_Module_API_Manual.html)。

### 9.6 方案 E：全屏 16:9 预览 + 4:3 检测（不建议，除非必须保留 4:3）

做法：ch0 使用 1920×1080 全屏预览，ch1 保留 320×240 检测。由于两通道宽高比不同，必须明确由传感器配置保证两者使用同一裁剪窗口；否则画面内容并不一一对应，检测框会偏离视频目标。

若硬件配置为“从同一 4:3 视野中裁剪中央 16:9 区域用于显示”，映射应为：

```text
4:3 检测图：320×240
显示区域对应中央 320×180
裁剪偏移：crop_y = (240-180)/2 = 30
x_screen = 6 * x_detect
y_screen = 6 * (y_detect - 30)
```

仅当 `30 ≤ y_detect < 210` 时，检测点才在显示画面内。此方案容易因通道裁剪策略不一致而产生叠加偏差，优先级低于方案 A/C/D。

### 9.7 OSD 与帧率策略

无论采用方案 A～E，遵守以下规则：

1. 视频预览通过 `Display.bind_layer()` 硬件直显，目标为 60 FPS；不要把 1920×1080 视频逐帧交给 `Display.show_image()`。
2. cv_lite 保持独立低分辨率通道运行，按实测帧率检测。
3. OSD 只绘制框、十字、状态文字；不要每帧创建新的 1920×1080 ARGB 图。
4. OSD 可限制在 10～15 FPS，或仅在检测结果变化时刷新；视频仍保持 60 FPS。
5. `to_ide=True` 的 1080P IDE 回传仅用于调试。官方示例提示高分辨率 IDE 预览可能偶发闪烁；实际 HDMI 演示优先关闭 IDE 回传。来源：选自 [CanMV 官方 Display 示例](https://www.kendryte.com/k230_canmv/en/main/example/media/display.html)。

### 9.8 建议的实施顺序

1. 对当前 640×480 屏幕，优先实现方案 A：`640×480` 硬件显示通道 + `320×240` 检测通道，验证完整 4:3 视野和 `×2` 坐标映射。
2. 禁止为了填满当前屏幕而采用非等比拉伸；圆环、矩形和靶心几何关系必须保持不变。
3. 检测图的模糊只用于抗噪，不能作为性能优化手段：模糊不减少像素数，反而增加运算。性能优化应依赖 ISP/硬件下采样与低分辨率检测通道；当前 `cv_lite` 已有 `GAUSSIAN_BLUR=5` 参数，需仅按实机噪声调节。
4. 只有接入外部 1920×1080 HDMI 屏时，才在方案 C/D/E 中选择；若首要目标是最大 4:3 视野，优先方案 C，若首要目标是 16:9 无黑边全屏，才选择方案 D。
5. 在每种模式下实测：视频 FPS、检测 FPS、OSD 刷新率、坐标叠加误差、靶纸进入边缘时的漏检情况。
