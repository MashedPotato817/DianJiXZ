# 差速运动学、系统架构与调试 —— Q&A

> 基于 WHEELTEC C07A / MSPM0G3507 实物代码分析

## Q1：差速小车怎么转弯？"运动学逆解"是什么？

### 差速原理

两轮差速小车只有一个自由度：**左右轮速度差产生旋转**。

```
        ┌─────────────┐
        │             │
左轮──→ │   车体      │ ←──右轮
  V_L   │             │   V_R
        └─────────────┘
         ← 轮距 W=0.13m →

线速度 Vx = (V_L + V_R) / 2         # 两轮平均速度 = 直线速度
角速度 Vz = (V_R - V_L) / W         # 速度差/轮距 = 旋转速度
```

**直觉**：
- V_L = V_R → 直行
- V_L < V_R → 左转（右轮快）
- V_L = -V_R → 原地旋转

### 逆解：目标 → 轮速

巡线算法算出的是 `(线速度Vx, 角速度Vz)`，需要转成左右轮分别的目标速度：

```c
// control.c:232-242
void Get_Target_Encoder(float Vx, float Vz)
{
    if (Vx < 0) Vz = -Vz;    // 倒车时转向方向反转

    MotorA.Target_Encoder = Vx - Vz * Wheelspacing / 2.0f;  // 左轮目标
    MotorB.Target_Encoder = Vx + Vz * Wheelspacing / 2.0f;  // 右轮目标
}
```

这是**运动学逆解**（Inverse Kinematics）：从期望的运动反推每个电机该怎么转。

### 举例

目标：向前 0.3 m/s，同时左转 2.0 rad/s：

```
左轮目标 = 0.3 - 2.0 × 0.13/2 = 0.3 - 0.13 = 0.17 m/s
右轮目标 = 0.3 + 2.0 × 0.13/2 = 0.3 + 0.13 = 0.43 m/s
```

右轮跑得比左轮快 → 车左转。

## Q2：编码器脉冲怎么换算成实际速度？

### 物理参数

| 参数 | 值 | 出处 |
| ---- | -- | ---- |
| 编码器线数 | 13 pulses/rev | `control.h` |
| 倍频方式 | 仅上升沿 → 2 倍频 | 硬件中断配置 |
| 减速比 | 30:1 | 代码注释 |
| 轮周长 | 0.2105 m | `control.h:Perimeter` |
| 控制频率 | 200 Hz（5ms 周期） | `control.h:Frequency` |

### 换算

```c
// control.c:217-229
MotorA.Current_Encoder = Encoder_A_pr × Frequency × Perimeter / 780.0f;
//                        ──────────   ────────   ────────   ───
//                        脉冲数       200Hz      0.21m     13×2×30
```

分母 `780 = 13 × 2 × 30`：
- 13 → 编码器线数
- 2 → 倍频
- 30 → 减速比

**物理含义**：电机轴转一圈 → 30 圈轮子 → `13×30 = 390` 个脉冲（单路不倍频）。`×2` 后变成 780。

把 5ms 内收到的脉冲数直接除以 780 再乘频率和周长，就得到了 m/s。

## Q3：主循环和中断是怎么配合的？为什么这样设计？

### 整体架构

```
main()
├── SYSCFG_DL_init()         # 一次性：外设初始化
├── OLED_Init()
│
└── while(1)                 # 主循环，一直跑
    ├── 读电池电压
    ├── 蓝牙数据处理
    ├── oled_show()          # OLED 刷新
    └── APP_Show()           # DataScope 上位机显示

═══════════════════════════════════════
           中断线（5ms一次）
═══════════════════════════════════════

TimerG 5ms中断
├── Key()                   # 按键扫描
├── LED_Flash()
├── 编码器读数 → 清零
├── Gray_Mode() / Get_RC()  # 巡线计算 / 遥控
├── PI_L / PI_R             # 速度控制
└── Set_PWM()               # 输出
```

### 为什么计算全放在中断里？

1. **定时精度**：5ms 一个控制周期，不容许主循环的 OLED/蓝牙处理影响周期
2. **从传感器到执行器一气呵成**：读编码器 → 算巡线 → 算 PID → 写 PWM，都在 5ms 内完成，**控制延迟固定**
3. **主循环只管"慢活"**：显示、通信

### 这个设计的代价

中断里代码不能太重。如果加多了 K210 帧解析、复杂路径规划，中断时间超过 5ms，下一个周期就会被推迟。

超时的应对方案：
- 把重计算（路径规划）挪到主循环，只设目标值
- 中断里只做传感器→执行器的快速闭环

## Q4：怎么在线调试？怎么看变量？

### 方法一：OLED 显示

`show.c` 里的 `oled_show()` 在主循环跑，显示关键状态。当前显示项：
- 运行模式（Diff）
- 左右轮 PWM 值
- 左右轮目标速度和实际速度（mm/s）
- 运行/停止状态

### 方法二：DataScope 上位机

`show.c` 里的 `DataScope_DP.h` + `APP_Show()` 把变量通过串口发给 PC 上位机，实时绘制波形。适合调 PID 参数时看速度曲线。

### 方法三：printf 串口打印

最原始但最可靠。把关键变量打印到串口终端。

### 方法四（推荐在比赛中）：LED 状态码

在关键时刻（十字检测、丢线、跑完等）让 LED 闪烁不同的次数/频率，不用看电脑也能判断状态。

## Q5：比赛现场怎么快速调参？

### 关键参数清单

| 参数 | 位置 | 调什么 |
| ---- | ---- | ------ |
| `GRAY_BASE_SPEED_MM_S` | `control.h:35` | 直线速度 |
| `GRAY_CURVE_SPEED_MM_S` | `control.h:37` | 弯道速度 |
| `GRAY_STEER_GAIN` | `control.h:43` | 转向灵敏度 |
| `GRAY_MAX_ANGULAR_SPEED` | `control.h:44` | 最大转角 |
| `Velocity_KP` | `control.c:30` | 速度响应快慢 |
| `Velocity_KI` | `control.c:30` | 速度稳态精度 |
| `GRAY_CROSS_RUN_TICKS` | `control.h:51` | 十字直冲时长 |

### 调参优先级

1. **先调底盘稳定性**：不巡线，遥控模式，让车直行——左右不偏、定距准确
2. **再调巡线**：低速（200 mm/s）巡简单赛道，调 STEER_GAIN 直到不冲出
3. **最后提速**：逐步加大 BASE_SPEED，每加一档都跑完整一圈验证

## Q6：增量式 PI 的"积分饱和"问题是什么？代码怎么处理的？

### 积分饱和（Integral Windup）

传统位置式 PID 的隐患：

```c
积分项 += Ki × 误差 × dt;
```

如果启动时电机被卡住——误差长时间很大 → 积分项不断积累 → 松开后输出爆炸 → 车猛冲。

### 增量式的天然抗饱和

因为增量式只保留 `Pwm` 一个累积量，且有：

```c
if (Pwm > 7800) Pwm = 7800;
if (Pwm < -7800) Pwm = -7800;
```

PWM 碰到上限就不会再增，天然防止饱和。不需要额外写"抗积分饱和"代码。

## Q7：整个控制系统的数据流是怎样的？（一句话总结）

```
传感器读数 → 速度换算 → 巡线算法 → 运动学逆解 → PI速度闭环 → PWM输出 → 电机
   ↑                                                                    │
   └────────────────── 编码器反馈 ─────────────────────────────────────┘
```

全部在 **5ms 中断** 内完成一个循环。主循环只做显示和通信。
