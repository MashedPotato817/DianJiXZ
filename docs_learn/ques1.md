# 传感器信号类型与 MCU 接口 —— Q&A

> 基于 WHEELTEC C07A / MSPM0G3507 实物代码分析

---

## Q1：传感器传给 MCU 的是数字量还是模拟量？分别是什么形式？

**答：三种形式都有，取决于传感器类型。**

| 传感器 | 原始信号 | 传入 MCU 的形式 | MCU 外设 | 最终数据类型 |
| ------ | -------- | --------------- | -------- | ------------ |
| 灰度 CCD | 模拟电压 | 模拟电压 | ADC | 128 个 0~255 整数 |
| 编码器 | 方波脉冲 | 数字电平（0/1） | GPIO 中断 | 脉冲计数值 |
| 电池电压 | 模拟电压 | 模拟电压（经分压） | ADC | 浮点数（伏特） |
| K210 视觉 | 串行帧 | 数字串行数据 | UART | 帧协议解析结果 |
| HTPA32 热成像 | 串行帧 | 数字串行数据 | UART / I2C | 温度数组 / 摘要 |

## Q2：灰度 CCD 传感器是怎么工作的？数据怎么到 MCU？

**传感器型号：TSL1401 线阵 CCD（128 像素）**

工作流程：

```
光照 → 128个光电二极管 → 积分（曝光）→ 模拟电压序列 → MSPM0 ADC → 12位数字值 → 软件压缩到 0~255
```

MCU 通过两根 GPIO 控制 CCD 的 CLK 和 SI 引脚，逐个像素读出模拟电压，然后走 MSPM0 内部 12 位 ADC 转换成数字量。

关键代码（`CCD.c:76-85`）：

```c
for (int i = 0; i < 128; i++)       // 读 128 个像素
{
    TSL_CLK(0);                     // 拉低CLK
    delay_us(100);                  // 曝光时间

    ADV[i] = (adc_getValue()) >> 4; // ADC采样 → 右移4位 → 0~4095 压缩到 0~255

    TSL_CLK(1);                     // 拉高CLK，切换到下一个像素
    delay_us(20);
}
```

**为什么右移 4 位？**

- MSPM0 ADC 是 12 位精度：0 ~ 4095
- 右移 4 位等价于 `÷16`，得到 0 ~ 255，刚好一个 `uint8_t`，节省内存且够用

**CCD 怎么变成"黑线位置"？**

拿到的 128 个灰度值并不能直接用——还需要三步：

1. **动态阈值**：取 128 个值的最大值和最小值，平均值作为阈值（`CCD.c:112`）
2. **找跳变沿**：从左往右找 "白→黑" 跳变（左边线），从右往左找 "黑→白" 跳变（右边线）
3. **算中点**：`中点 = (左边线位置 + 右边线位置) / 2`，这就是黑线在 CCD 视野里的像素坐标（`CCD_Zhongzhi`，0~127）

最后用这个中点偏差做 PD 巡线控制（`CCD.c:14-17`）：

```c
Bias = 64 - CCD_Zhongzhi;   // 64是CCD中心，偏差=实际位置减中心
Move_Z = Bias * KP + (Bias - Last_Bias) * KI;  // PD控制
```

## Q3：编码器的信号是什么样的？MCU 怎么"读"？

### 物理信号

编码器是**正交编码器**——两个霍尔传感器（A 相 / B 相）输出两路方波，相位差 90°：

```
正转：
     A ──┐   ┌───┐   ┌───
         └───┘   └───┘
     B    ┌───┐   ┌───┐
      ────┘   └───┘   └───
           ↑A领先B 90°

反转：
     A    ┌───┐   ┌───┐
      ────┘   └───┘   └───
     B ──┐   ┌───┐   ┌───
         └───┘   └───┘
           ↑B领先A 90°
```

### MCU 怎么接收

MSPM0 用 **GPIO 边沿中断** 捕获。A/B 相的每个上升沿都触发中断，中断里读另一相的电平判断方向：

```c
// encoder.c:18-28
if (A相上升沿触发) {
    if (B相 == 低电平)  计数--;  // 反转
    else               计数++;   // 正转
}
```

这就是 **4 倍频判向**——每个脉冲周期能检测 4 次变化（A 升、A 降、B 升、B 降），分辨率是编码器线数的 4 倍。

### 编码器参数与速度换算

你的小车编码器参数（`control.h`）：

| 参数 | 值 | 含义 |
| ---- | -- | ---- |
| `ENCODER_LINES` | 13 | 每转 13 个脉冲 |
| `MULTIPLY_FACTOR` | 2 | 2 倍频（仅上下边沿） |
| `GEAR_RATIO` | 28 | 减速比 28:1 |

10ms 中断里把脉冲计数换算成速度：

```c
速度(m/s) = 脉冲数 × 控制频率 × 轮周长 / (编码器线数 × 倍频 × 减速比)
          = count × 100 × 0.21 / (13 × 2 × 28)
          = count × 100 × 0.21 / 728
```

**注意**：这里用的是 2 倍频而不是中断里看到的 4 倍频——因为代码里只响应了上升沿中断（`DL_GPIO_PIN_x_EDGE_RISE`），下降沿没开。

## Q4：MSPM0 的 ADC 怎么把模拟电压变成数字？

MSPM0G3507 内置 12 位逐次逼近型（SAR）ADC，转换流程：

```
模拟输入引脚 → 采样保持 → 逐次比较 → 12位结果寄存器 → 软件读取
```

三个关键参数：
- **参考电压**：3.3V
- **分辨率**：12 位 → 4096 级
- **转换时间**：取决于 ADC 时钟，SAR 型通常几个 μs

电压换算公式：

```c
实际电压 = ADC读数 × 3.3 / 4096
```

例如 ADC 读到 2048 → `2048 × 3.3 / 4096 = 1.65V`

## Q5：数字量传感器（K210 / 热成像）是怎么通信的？

不在当前固件中，但按规划：

**K210 视觉模块 → UART**

```
K210（运行识别模型）→ UART 发帧 → MSPM0 UART1 接收（9600bps, DMA）
```

帧格式通常类似：
```
帧头(0xAA) + 命令字 + 数据长度 + 数据(识别到的颜色/坐标/类型) + 校验
```

MSPM0 侧：DMA 收帧 → 环形缓冲区 → 主循环解析

**HTPA32 热成像 → I2C 或 UART**

低速传感器，输出 32×32 温度矩阵或摘要（最高温、最低温、中心温、最高温坐标）。MSPM0 当 I2C 主设备主动读取，或通过 UART 被动接收帧。

## Q6：中断里能做什么、不能做什么？

**架构**：10ms 定时器中断是控制核心，它串联了所有传感器的数据流：

```
TimerG中断(10ms)
  ├── 读编码器脉冲累计值 → 清零
  ├── 从当前脉冲数换算速度（m/s）→ 低通滤波
  ├── 读CCD灰度值 → 算中线偏差 → PD巡线控制
  ├── 速度控制（增量式PI）
  ├── Set_PWM() 输出
  └── LED闪烁 / 按键扫描
```

**GPIO中断**：只做编码器 ±1 计数，不做任何计算

这个架构保证了 10ms 控制周期严格一致，所有计算都在一个中断里完成。潜在问题是如果计算量太大（比如加了 K210 帧解析、图像处理），中断执行时间超过 10ms 就会丢周期——需要把重计算挪到主循环。
